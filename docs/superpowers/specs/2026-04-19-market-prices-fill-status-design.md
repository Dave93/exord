# Статус заказа «Заполнение цен базара»

**Дата:** 2026-04-19
**Статус:** Design approved

## Цель

Добавить в workflow обычных заказов отдельный статус «Заполнение цен
базара», чтобы после закрытия заказа оператор мог ввести фактические
суммы, за которые были куплены базарные (рыночные) позиции, и только
после этого заказ переходил бы в «Завершён».

## Термины

- **Базарная позиция** — позиция заказа `order_items`, продукт которой
  привязан к группе товаров `product_groups` с флагом `is_market = 1`
  (через `product_groups_link`).
- **Обычный заказ** — запись в `orders` c `is_market = 0 OR NULL`
  (в отличие от базарных заказов, которые обрабатываются
  `MarketOrdersController`).

## Workflow

```
0 Новый → actionSend → 1 Отправлен → actionClose →
    ├── есть базарные позиции → 4 Заполнение цен базара
    │       └── actionMarketPricesSave (все цены заполнены) → 2 Завершён
    └── нет базарных позиций → 2 Завершён (как сейчас)
```

Отдельный статус «Приёмка» не добавляем — ввод факта продолжает
выполняться через существующие `actionFactStock` / `actionFactSupplier`,
которые заполняют `factStoreQuantity` / `factSupplierQuantity` и
не меняют статус.

## Изменения схемы БД

Миграция:

- `orders.state` — добавляем значение `4`, тип не меняется.
- `order_items` — добавить колонку:

  ```sql
  market_total_price DECIMAL(12,2) NULL
  ```

  Общая сумма, за которую куплена позиция (total, а не цена за единицу).
  `NULL` означает, что цена ещё не введена. Существующее поле `price`
  не переиспользуем — оно занято и имеет другую семантику (цена за
  единицу, используется в `OrdersController`, `MarketOrdersController`,
  `ApiController`).

В `models/Orders.php`:

```php
public static $states = [
    0 => "Новый",
    1 => "Отправлен",
    2 => "Завершен",
    3 => "Проверка офисом",
    4 => "Заполнение цен базара",
];
```

## Переход в статус 4 при закрытии

`OrdersController::actionClose($id)` — после успешного закрытия заказа
в iiko (блок, где сейчас ставится `state = 2`):

1. Определить, содержит ли заказ хотя бы одну базарную позицию.
   Добавить метод `Orders::hasMarketItems(): bool`:

   ```php
   public function hasMarketItems()
   {
       return (new Query())
           ->from('order_items oi')
           ->innerJoin('product_groups_link pgl', 'pgl.productId = oi.productId')
           ->innerJoin('product_groups pg', 'pg.id = pgl.productGroupId')
           ->where([
               'oi.orderId' => $this->id,
               'pg.is_market' => 1,
               'oi.deleted_at' => null,
           ])
           ->exists();
   }
   ```

2. Если метод возвращает `true` → `state = 4`.
   Иначе → `state = 2` (текущее поведение).

Telegram-уведомление о закрытии и логирование в `iiko` категорию
сохраняем как есть. Для перехода в `4` отправляется отдельное
уведомление — «Заказ ожидает заполнения цен базара» — тем же
каналом. (Текст и chat_id уточнить при имплементации; если нет явного
требования — оставить то же самое уведомление, что и при закрытии.)

## Раздел «Заполнение цен базара»

### Действие и роут

`OrdersController::actionMarketPrices()`:

- Список заказов с `state = 4` и `deleted_at IS NULL`.
- Фильтры: дата, филиал (как в существующем `actionStockOrders`).
- Аналогично существующим спискам использовать `OrderSearch`.
- Колонки: № заказа, Филиал, Дата, Дата закрытия, Кол-во базарных
  позиций, Сколько из них с заполненной `market_total_price`, Сумма
  уже введённых цен.

### Доступ

`AccessRule` в `OrdersController::behaviors()` для
`actionMarketPrices` и `actionMarketPricesFill`:

- Разрешённые роли: `ROLE_ADMIN`, `ROLE_OFFICE`, `ROLE_OFFICE_MANAGER`.

Пункт меню «Заполнение цен базара» добавить в `views/layouts/panel.php`
(или соответствующий layout-файл — определить при имплементации),
видимый тем же ролям.

### Вьюха `views/orders/market-prices.php`

GridView с `DataProvider` из `OrderSearch` с форсированным
`state = 4`. Ссылка на каждый заказ открывает
`orders/market-prices-fill?id=...`.

## Страница заполнения цен

### Действие `actionMarketPricesFill($id)`

1. Находим заказ; если `state != 4` — редирект на список с flash-сообщением.
2. Тянем базарные позиции через
   `Orders::getOrderProducts($id, true)` (существующий метод —
   второй параметр `$is_market = true` уже фильтрует по
   `pg.is_market = 1`).
3. `GET` — рендерим вьюху с формой.
4. `POST`:
   - Параметр `action = 'save'` → сохранить введённые цены,
     остаться в статусе `4`, flash «Сохранено», редирект на ту же
     страницу.
   - Параметр `action = 'finish'` → валидация: у **всех** базарных
     позиций заказа `market_total_price > 0`. Если нет — ошибка
     с перечислением незаполненных позиций, остаёмся в статусе `4`.
     Если да — сохраняем, `state = 2`, пишем в логи, редирект на
     `orders/view?id=...`.

Сохранение значений:

```php
foreach ($post['Prices'] as $productId => $value) {
    if ($value === '' || $value === null) {
        continue;
    }
    $oi = OrderItems::findOne(['orderId' => $model->id, 'productId' => $productId]);
    if ($oi === null) continue;
    $oi->market_total_price = (float)$value;
    $oi->save(false, ['market_total_price']);
}
```

### Вьюха `views/orders/market-prices-fill.php`

- Шапка: № заказа, филиал, дата заказа, дата закрытия, комментарий.
- Таблица с базарными позициями. Колонки:
  - Продукт (с группой для справки)
  - Ед. изм. (`mainUnit`)
  - Кол-во заказа (`quantity`)
  - Факт приёма (`factStoreQuantity` / `factSupplierQuantity`, либо
    тот, что ненулевой — см. существующий `view` заказа)
  - **Сумма за позицию** — `<input type="number" step="0.01">`,
    привязан к `Prices[productId]`, начальное значение из
    `market_total_price`.
- Под таблицей: итоговая сумма (live-подсчёт JS + server-side).
- Две кнопки: «Сохранить» и «Сохранить и завершить».

## Отображение в существующих местах

- `orders/view` — если `state = 4` и у пользователя есть доступ,
  показать кнопку «Заполнить цены базара» → ссылка на
  `orders/market-prices-fill?id=...`.
- Общие списки заказов (там где выводится `Orders::$states[$state]`) —
  новый статус будет показываться автоматически. Проверить фильтры
  по статусу, чтобы `4` был доступен как опция, если это нужно
  (при имплементации — быстрый поиск по `OrderSearch::$state`).
- Условие `orders.state < 2` в `OrdersController::actionStock` (строка
  917: `where orders.state<1 ... (is_market=0 or is_market is null)`)
  и похожие места — проверить, не должен ли `state = 4` исключаться
  из них наряду с `state = 2`. Ожидание: `state = 4` — это «после
  закрытия», поэтому в списки текущей работы со складом попадать
  не должен. При имплементации просмотреть все места, где
  фильтрация по `state` и при необходимости добавить `state != 4`.

## Не в скоупе

- Отдельный статус «Приёмка» между 1 и 4 (решено в брейнсторме: нет).
- Изменения логики `MarketOrdersController` (отдельные базарные
  заказы) — их эта фича не касается.
- Отчёты и аналитика по суммам `market_total_price` — отдельной
  задачей, если понадобится.
- Многократный запуск iiko-документов при возврате в статус 4 —
  предполагается, что `actionClose` вызывается один раз.
  Откат/повторный переход между `4` и `2` не описан, будет решён,
  если возникнет потребность.
